# SignalDesk 系统设计

## 1. 技术选型

- 后端：`Java 17 + Spring Boot 3 + Maven`
- 前端：`Vue 3 + Vite + TypeScript + Pinia + Vue Router`
- 数据库：`MySQL 8`
- 缓存：`Redis`
- 搜索：`Elasticsearch / OpenSearch`

## 2. 架构风格

SignalDesk 采用模块化单体架构。

这样做的原因是：

- 对个人项目更友好，复杂度可控
- 便于快速开发和调试
- 更适合作为秋招展示项目
- 后续如果需要，也可以按模块拆分

## 3. 后端模块

- `auth`：登录注册、JWT 鉴权
- `user`：用户信息
- `source`：信息源管理
- `document`：抓取结果与内容归档
- `crawltask`：抓取任务调度与执行记录
- `search`：全文搜索
- `summary`：摘要生成与引用管理
- `dashboard`：统计数据与最近变化
- `audit`：审计日志
- `infrastructure`：公共配置、异常、响应封装

## 4. 核心流程

### 4.1 抓取流程

1. 用户手动触发或调度器自动触发抓取
2. 创建抓取任务记录
3. 拉取网页或目标内容
4. 对内容进行解析、去重和标准化
5. 写入文档表，并更新最新版本状态
6. 更新抓取历史
7. 同步到搜索索引

### 4.2 摘要流程

1. 用户对信息源或搜索结果发起摘要请求
2. 系统收集最近文档内容
3. 调用 AI 模块生成摘要
4. 保存摘要和引用来源
5. 前端展示摘要内容与引用关系

### 4.3 搜索流程

1. 用户输入关键词
2. 后端调用搜索模块查询索引
3. 返回高亮片段和来源信息
4. 前端展示结果并支持跳转到信息源详情

## 5. 设计原则

- 优先完成 P0 主流程
- AI 是增强能力，不是系统基础依赖
- 模块边界清晰，避免职责混乱
- 失败状态必须可见、可排查
- 搜索和摘要都要围绕“可展示、可演示”设计

## 6. 系统边界

### 系统内负责

- 用户认证与工作空间
- 信息源管理
- 抓取任务管理
- 内容保存与展示
- 搜索与摘要
- 前端交互体验

### 系统外依赖

- MySQL：主数据存储
- Redis：缓存与短期状态
- Elasticsearch / OpenSearch：搜索索引
- 第三方网页与 RSS：被监控内容来源
- OpenAI 或兼容服务：摘要生成

## 7. 前后端交互

前端通过 REST API 与后端通信，统一响应格式为：

```json
{
  "code": 0,
  "data": {},
  "message": "ok"
}
```

接口范围主要包括：

- 认证接口
- 信息源接口
- 抓取任务接口
- 文档接口
- 搜索接口
- 摘要接口
- 仪表盘接口

## 8. 部署建议

本地开发推荐：

- 前端：`npm run dev`
- 后端：`mvn spring-boot:run`
- MySQL / Redis / Elasticsearch：本机或 Docker

后续演示推荐：

- 使用 `Docker Compose` 管理依赖服务
- 后端打包为 jar 或镜像
- 前端构建为静态资源后统一部署
