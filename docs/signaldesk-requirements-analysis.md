# SignalDesk 需求分析

## 1. 产品定位

SignalDesk 是一个面向个人学习、求职跟踪和轻量研究场景的信息监控平台。  
用户可以把网站、RSS、PDF 和手动笔记统一纳入一个工作台，定时抓取、保存历史内容、搜索旧资料，并基于抓取结果生成摘要。

它的定位介于“简单收藏工具”和“复杂企业知识平台”之间。  
第一版强调的是：

- 实用
- 清晰
- 可演示
- 适合作为开源作品集项目

## 2. 目标用户

### 核心用户

- 准备实习或秋招的学生
- 需要持续跟踪公司招聘页、技术博客、学习资料的人
- 想把零散信息集中管理的个人用户

### 次级用户

- 独立开发者
- 小团队成员
- 需要跟踪竞品、文档更新或研究资料的用户

## 3. 核心问题

SignalDesk 主要解决以下问题：

1. 重要信息分散在网页、PDF、收藏夹和笔记里，不方便统一管理
2. 手动回访信息源效率低，容易错过更新
3. 保存下来的内容缺少统一搜索入口
4. 纯 AI 总结缺少来源支撑，不够可信
5. 现有工具要么太轻，要么太重，不适合个人场景

## 4. 产品目标

第一版至少要做到：

- 能添加和管理信息源
- 能手动抓取和定时抓取
- 能保存抓取历史和最新内容
- 能在前端查看信息源详情
- 能搜索历史内容
- 能生成带来源依据的摘要

## 5. V1 非目标

第一版暂不覆盖以下内容：

- 企业级多租户和计费体系
- 复杂工作流编排市场
- 浏览器插件生态
- 细粒度团队权限系统
- 多人实时协作编辑

## 6. 功能范围

### P0

- 登录与注册
- 个人工作空间
- 信息源管理：URL、RSS、PDF、手动笔记
- 标签管理
- 手动抓取与定时抓取
- 抓取历史查看
- 最近内容查看
- 仪表盘
- 搜索与筛选
- AI 摘要与引用来源
- 信息源详情页

### P1

- 网页更新 diff 视图
- 周报或阶段摘要
- 保存搜索主题
- 置顶重要信息源
- 失败任务重试队列
- 审计日志

### P2

- 团队空间
- 更复杂的角色权限系统
- 消息通知集成
- 可视化工作流编排

## 7. 关键用户故事

### 用户故事 1

作为一名学生，我希望添加多个招聘页作为信息源，这样我不用每天手动打开官网，也能知道岗位是否有变化。

### 用户故事 2

作为一名用户，我希望能搜索自己保存过的网页、PDF 和笔记内容，这样我可以快速找到之前看过的资料。

### 用户故事 3

作为一名用户，我希望系统能对最近抓取到的新内容生成摘要，这样我可以更快浏览重点信息。

### 用户故事 4

作为一名用户，我希望查看某个信息源的详情页，了解它最近什么时候抓取过、抓到了什么、有没有失败记录。

## 8. 页面建议

建议至少包含以下页面：

1. 登录页
2. 注册页
3. 仪表盘
4. 信息源列表页
5. 新建/编辑信息源页
6. 信息源详情页
7. 搜索页
8. 摘要历史页

## 11. 技术方案建议

- 后端：`Java 17 + Spring Boot 3 + Maven`
- 数据库：`MySQL 8`
- 缓存：`Redis`
- 搜索：`Elasticsearch` 或 `OpenSearch`
- 前端：`Vue 3 + Vite + TypeScript + Pinia + Vue Router`
- 部署：优先 `Docker Compose`

## 12. 项目风险

- 需求范围膨胀
- 搜索和摘要模块复杂度过高
- 抓取流程面对复杂站点时不稳定
- 前端容易做成普通后台，而不是有展示感的产品

## 13. 简历价值

这个项目适合作为秋招作品，因为它同时覆盖：

- Java 后端工程能力
- Spring Boot 模块化设计
- Maven 工程管理
- 数据建模能力
- 定时任务与异步流程
- 搜索系统集成
- AI 摘要能力
- Vue 前端页面与交互设计

## 14. 推荐开发节奏

### 第 1-2 周

- 明确数据模型和模块边界
- 完成登录注册和信息源 CRUD

### 第 3-4 周

- 完成抓取流程和抓取历史
- 打通信息源详情页

### 第 5-6 周

- 完成搜索和摘要主流程
- 补仪表盘和交互细节

### 第 7-8 周

- 打磨前端体验
- 补测试、README、截图和演示脚本
